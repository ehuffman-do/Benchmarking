# Day-2 Operations Catalog — Percona Operator for PostgreSQL v2 (perconapgcluster) & Crunchy PGO v5 (postgrescluster)

Verified against: **Percona operator docs source @ 2026-06-03 (covers v2.8.x / pre-2.9)** (github.com/percona/k8spg-docs, commit 391b5708), **percona/percona-postgresql-operator main** (Go API types + deploy/cr.yaml), **CrunchyData/postgres-operator v5.8.8 source** (API types, `internal/naming/annotations.go`, `internal/naming/names.go`, pgupgrade controller), plus Crunchy access.crunchydata.com docs via search. Percona v2 is a fork of PGO v5: **it keeps the upstream `postgres-operator.crunchydata.com/*` labels and most annotations**, and layers `pgv2.percona.com/*` CRDs on top. This means one console codepath can share ~80% of mechanics between the two operators.

## 0. Shared object model (needed by every operation)

**CRDs**
- Percona: `perconapgclusters.pgv2.percona.com/v2` (kind `PerconaPGCluster`, shortname **`pg`**), `perconapgbackups` (shortname **`pg-backup`**), `perconapgrestores` (shortname **`pg-restore`**), `perconapgupgrades` (kind `PerconaPGUpgrade`).
- Crunchy: `postgresclusters.postgres-operator.crunchydata.com/v1beta1` (kind `PostgresCluster`), `pgupgrades` (kind `PGUpgrade`). No separate backup/restore CRDs — backups/restores are driven by spec sections + annotations.

**Pod anatomy (identical in both, from `internal/naming/names.go`)**
- Instance pod (`<cluster>-<instanceSet>-<rand4>-0`, one single-replica StatefulSet per Postgres member): init `postgres-startup`; containers `database` (postgres + patroni), `replication-cert-copy`, `pgbackrest`, `pgbackrest-config`; optional `exporter` (Crunchy pgMonitor), `pmm-client` (Percona PMM). Annotation `kubectl.kubernetes.io/default-container: database` is set, so bare `kubectl logs` hits the database container.
- Repo host pod `<cluster>-repo-host-0`: containers `pgbackrest`, `pgbackrest-config`.
- Pooler pods: Deployment `<cluster>-pgbouncer`, containers `pgbouncer`, `pgbouncer-config`.

**Labels** (both operators): `postgres-operator.crunchydata.com/cluster=<name>`, `.../instance` (member), `.../instance-set`, `.../role` (`master`|`replica`), `.../data=postgres|pgbackrest`, `.../pgbackrest-backup` (backup Jobs), `.../pgbackrest-cronjob`, `.../pgbackrest-restore`, `.../crunchy-postgres-exporter`.

**Services**: `<cluster>-primary` (RW), `<cluster>-replicas` (RO), `<cluster>-ha` (Patroni leader service + the Kubernetes-DCS Endpoints object), `<cluster>-pods`, `<cluster>-pgbouncer`. Patroni scope (cluster name inside patronictl) = **`<cluster>-ha`**. Patroni REST on port **8008** (`/liveness`,`/readiness`,`/cluster`,`/patroni`), Postgres 5432, exporter 9187.

**Cluster status**
- Percona `status.state`: `initializing` | `ready` | `paused` | `stopping`; `status.postgres.{size,ready,instances[{name,size,ready}],version,imageID}`; `status.pgbouncer.{size,ready}`; `status.pgbackrest` (inlined Crunchy status); `status.patroni`.
- Crunchy `status.instances[{name,replicas,readyReplicas,updatedReplicas,desiredPGDataVolume}]`, `status.pgbackrest.{manualBackup,scheduledBackups[],restore,repos[]}`, `status.observedGeneration`, `status.conditions` with known types: `PersistentVolumeResizing`, `PersistentVolumeResizeError`, `Progressing`, `ProxyAvailable`, plus pgBackRest conditions `PGBackRestManualBackupSuccessful`, `PostgresDataInitialized`, `PGBackRestReplicaCreate`, `PGBackRestReplicaRepoReady`.
- Generic completion idiom for any declarative change: `status.observedGeneration == metadata.generation` AND ready==size (Percona: `status.state==ready`).

**kubectl-only console patterns**: `kubectl patch <kind> <name> --type=merge -p '<json>'` (or `--type=json` for array index ops), `kubectl annotate --overwrite`, `kubectl exec <pod> -c database -- patronictl list -f json` (machine-readable!), `kubectl exec ... -- psql -c '...' -At -F,`, `kubectl logs -c <container> --since= --previous`, `kubectl get events --field-selector involvedObject.name=<x> --sort-by=.lastTimestamp`, `kubectl wait --for=condition=...`.

## Frequency ranking (how often real operators do these)

Tier A (daily/continuous): 1 replication & cluster health inspection, 2 log inspection, 3 backup status/verification + on-demand backup.
Tier B (weekly/monthly): 4 restart (cluster/instance), 5 PostgreSQL parameter changes (incl. autovacuum tuning), 6 switchover, 7 user/db/password management, 8 scale replicas, 9 vertical resize, 10 backup schedule/retention edits.
Tier C (quarterly): 11 minor version upgrade, 12 PVC expansion, 13 pooler ops, 14 node maintenance, 15 clone to new cluster.
Tier D (rare, high stakes): 16 restore/PITR, 17 failover (unplanned), 18 pause/shutdown/unmanaged, 19 pg_repack/bloat intervention, 20 extension management, 21 cert rotation, 22 monitoring enable/change, 23 major version upgrade, 24 standby promotion (DR).

## 1. Replication & cluster health inspection (Tier A)
- **What**: read-only status of Patroni topology, lag, slots, archiver.
- **Mechanics**: `kubectl exec <any-instance-pod> -c database -- patronictl list -f json` (columns: Member, Host, Role Leader/Replica, State running/streaming/`in archive recovery`/crashed, TL timeline, "Lag in MB"); `patronictl history` (failover history); `curl -s localhost:8008/cluster` inside pod. SQL on primary: `SELECT application_name,state,sync_state,write_lag,flush_lag,replay_lag, pg_wal_lsn_diff(pg_current_wal_lsn(),replay_lsn) AS lag_bytes FROM pg_stat_replication;` and `SELECT slot_name,slot_type,active,pg_wal_lsn_diff(pg_current_wal_lsn(),restart_lsn) AS retained_wal,safe_wal_size FROM pg_replication_slots;` and `SELECT archived_count,failed_count,last_failed_time FROM pg_stat_archiver;`. On replica: `SELECT pg_is_in_recovery(); SELECT status,received_lsn FROM pg_stat_wal_receiver;`. Role via labels: `kubectl get pods -l postgres-operator.crunchydata.com/cluster=<c> -L postgres-operator.crunchydata.com/role -L postgres-operator.crunchydata.com/instance`.
- **Key caveat**: default Patroni config here sets `use_slots: false` (verified in Percona `patronictl show-config` output) — physical streaming does NOT use slots; replicas catch up via streaming or pgBackRest archive. But user/CDC **logical** slots exist and an **inactive slot retains WAL forever → PVC-full outage**. Surface any `active=false` slot with retained_wal, and recommend `max_slot_wal_keep_size` (PG13+, `safe_wal_size` column goes negative-ish/NULL as danger indicator).
- **Risk**: none (read-only). **Declarative**: n/a. **Completion**: n/a. **Live signals**: all the above, refreshed 2–5s.

## 2. Log inspection (Tier A)
- **Mechanics**: `kubectl logs <pod> -c database` (postmaster stderr + Patroni), `-c pgbackrest`, `-c replication-cert-copy` (TLS/replication cert issues), `-c pgbouncer`, `-c exporter`/`-c pmm-client`; flags `-f`, `--since=600s`, `--previous`, `--timestamps`. PostgreSQL server logfiles (log_collector) live at **`/pgdata/pg<major>/log/`** — `kubectl exec ... -c database -- sh -c 'ls -t /pgdata/pg17/log | head; tail -n 200 /pgdata/pg17/log/<file>'` (csvlog if `log_destination=csvlog` set). pgBackRest logs: backup Job pod logs; repo-host `/pgbackrest/repo1/log`. Operator logs: `kubectl logs deploy/percona-postgresql-operator -n <op-ns>` / `deploy/pgo`. Events: `kubectl describe pg <c>` / `describe postgrescluster <c>`.
- **Risk**: none. **Live signals**: tail streams; grep presets: `FATAL|PANIC|ERROR`, `could not`, `archive-push`, `No space left`.

## 3. Backups: on-demand, verification (Tier A)
- **Percona (CRD-based, preferred)**: create `PerconaPGBackup`: `{apiVersion: pgv2.percona.com/v2, kind: PerconaPGBackup, metadata.name, spec:{pgCluster: cluster1, repoName: repo1, options:["--type=full"]}}` (options = any pgBackRest CLI flags, e.g. `--type=full|diff|incr`, `--log-level-stderr=debug`). Track: `kubectl get pg-backup <name>` → `status.state`: `""(new)→Starting→Running→Succeeded|Failed`; also `status.backupName` (pgBackRest label like `20240628-074416F`), `status.backupType`, `status.latestRestorableTime` (when `spec.backups.trackLatestRestorableTime` enabled). Cluster gets transient annotation `pgv2.percona.com/backup-in-progress`. Deleting the `pg-backup` object deletes the physical backup from the repo (finalizer `percona.com/delete-backup`); `percona.com/delete-backups` finalizer on the cluster removes all on cluster delete (tech preview).
- **Crunchy (spec+annotation)**: 1) set `spec.backups.pgbackrest.manual: {repoName: repo1, options: ["--type=full"]}`; 2) trigger: `kubectl annotate -n <ns> postgrescluster <c> --overwrite postgres-operator.crunchydata.com/pgbackrest-backup="$(date '+%F_%H:%M:%S')"` (new value = new backup). Completion: condition `PGBackRestManualBackupSuccessful=True`; `status.pgbackrest.manualBackup{id,finished,succeeded,failed,active}`; backup Job labeled `postgres-operator.crunchydata.com/pgbackrest-backup`.
- **Verification (both)**: `kubectl exec <instance-pod> -c database -- pgbackrest --stanza=db info` (or on repo-host `-c pgbackrest`), `pgbackrest --stanza=db check`.
- **Risk**: low (I/O load on source; `--start-fast=y`, `--process-max=N` tunables). **Declarative**: Percona fully (CR object); Crunchy semi (spec + imperative annotate). **Completion**: above. **Live signals**: Job active/succeeded, backup pod log tail, `pg_stat_archiver` counters, repo-host PVC usage (`kubectl exec <repo-host> -c pgbackrest -- df -h /pgbackrest`), DB write latency from pgbench harness.
- **Gotcha**: initial `replica-create` full backup is automatic on cluster creation and (Percona) hidden from `pg-backup` list; a cluster can't be paused while a backup Job runs (delete Job first: `kubectl delete job -l postgres-operator.crunchydata.com/pgbackrest-backup`).

## 4. Restart cluster / single instance (Tier B)
- **Whole-cluster rolling restart (both operators, documented for Crunchy)**: patch a dummy pod-template annotation: `kubectl patch postgrescluster/<c> --type merge -p '{"spec":{"metadata":{"annotations":{"restarted":"'"$(date)"'"}}}}'` — same path on `pg` for Percona (`spec.metadata.annotations` propagates to all pod templates → operator performs rolling recreation, replicas first, primary last ⇒ exactly one switchover blip). Instance-set-only: `spec.instances[i].metadata.annotations`. Do NOT use `kubectl rollout restart sts` (operator owns the STS and reverts).
- **Single member, in-place (no pod recreate)**: `kubectl exec <pod> -c database -- patronictl restart <cluster>-ha <member-pod-name> --force` — this is also the way to apply `pending_restart` parameters (see op 5). Or `kubectl delete pod <pod>` (Patroni re-elects if it was the leader).
- **Risk**: medium — primary restart = seconds of write unavailability; pgBouncer masks some of it.
- **Declarative**: yes for annotation method; no for patronictl/delete-pod.
- **Completion**: all pods AGE reset / `status.instances[].updatedReplicas==replicas`; Percona `status.state` returns `ready`; `SELECT pg_postmaster_start_time();` newer than trigger; patronictl list no `Pending restart` flag.
- **Live signals**: pod phase/restart-count watch, role-label flips, patronictl list loop, connection-error rate from harness.

## 5. PostgreSQL parameter changes incl. autovacuum tuning (Tier B)
- **Mechanics (both)**: `spec.patroni.dynamicConfiguration.postgresql.parameters.<name>=<value>` (kubectl patch merge). Patroni applies via its dynamic config; params with `postmaster` context (`shared_buffers`, `wal_level`, `max_connections`, `shared_preload_libraries`…) set `pending_restart` and **Patroni auto-performs a rolling restart** (Percona docs) — Crunchy PGO applies config and the restart happens on next Patroni-driven restart or `patronictl restart`. Check restart requirement first: `SELECT name,context FROM pg_settings WHERE name='<x>';`.
- Manual/emergency path: `kubectl exec ... -- patronictl edit-config --pg <name>=<value>` (reverted by operator on reconcile unless cluster `unmanaged`); ConfigMap override hatch: annotate `<cluster>-config` (or per-pod `<pod>-config`) with `pgv2.percona.com/override-config=true` (Percona; Crunchy equivalent `postgres-operator.crunchydata.com/override-config`), then edit CM + `patronictl reload` — must remove annotation afterwards.
- **Operator-managed immutable params** (reverted): `archive_command`, `archive_mode`, `restore_command`, TLS/ssl_* , `unix_socket_directories`, etc. `pg_hba` extra rules: `spec.patroni.dynamicConfiguration.postgresql.pg_hba[]` (appended; order matters).
- Autovacuum knobs to expose: `autovacuum_max_workers` (restart), `autovacuum_naptime`, `autovacuum_vacuum_cost_limit/_delay`, `autovacuum_vacuum_scale_factor`, `autovacuum_analyze_scale_factor`, `maintenance_work_mem`; per-table overrides via SQL `ALTER TABLE ... SET (autovacuum_vacuum_scale_factor=...)`.
- **Risk**: medium-high — the operator does **not validate values**; a bad `shared_preload_libraries` or oversize `shared_buffers` prevents postgres start cluster-wide (CrashLoop). Console must pre-validate against pg_settings registry.
- **Declarative**: yes. **Completion**: `patronictl show-config` reflects value; `SHOW <param>;` on each member; no `Pending restart` in `patronictl list`. **Live signals**: rolling-restart progress, pg_settings diff per member, error-log tail.

## 6. Switchover & failover (Tier B / D)
- **Mechanics (identical field names; Percona verified to use the Crunchy trigger annotation)**:
  1. `kubectl patch pg/<c> --type=merge -p '{"spec":{"patroni":{"switchover":{"enabled":true,"targetInstance":"<instance-name e.g. cluster1-instance1-bmdp>"}}}}'` — `targetInstance` optional for type Switchover (random target if omitted); **required** for `type: Failover`. Field `spec.patroni.switchover.type`: enum `Switchover` (default) | `Failover` (last-resort, forces primary regardless of health).
  2. Trigger: `kubectl annotate --overwrite pg/<c> postgres-operator.crunchydata.com/trigger-switchover="$(date)"` (same annotation for Crunchy `postgrescluster`).
  3. Afterwards set `enabled:false` (documented hygiene).
  - Note `targetInstance` is the **instance/StatefulSet name** (pod name minus `-0`).
  - Emergency imperative path: `kubectl exec <pod> -c database -- patronictl switchover <cluster>-ha --leader <cur> --candidate <target> --force` or `patronictl failover --candidate <target> --force`.
- **Risk**: medium (planned): seconds of RW interruption; Failover type: high (potential data loss on unreplicated txns).
- **Declarative**: yes (spec + annotation).
- **Completion**: Crunchy `status.patroni.switchover` == annotation value (and `switchoverTimeline` cleared); role labels flipped (`-L role` shows new `master`); `patronictl list` new Leader, timeline TL incremented; both replicas back to `streaming`.
- **Live signals**: 1–2s loop on `patronictl list -f json`; role labels; `pg_is_in_recovery()` per member; harness TPS dip/recovery; `pg_stat_replication` re-attach; timeline history (`patronictl history`).

## 7. User & database management, password rotation (Tier B)
- **Mechanics (same schema both)**: `spec.users[]: {name (req, DNS-1123, lowercase), databases: [..], options: "SUPERUSER CREATEDB ..." (ALTER ROLE attrs; ≤200 chars, no PASSWORD, no comments — CRD-validated), password: {type: ASCII|AlphaNumeric (default ASCII)}, secretName (Percona custom-secret ref), grantPublicSchemaAccess (Percona, PG≥15)}`. Databases in `users[].databases` are created; **removal never drops** users/DBs/role attrs (revoke with explicit `NOSUPERUSER` etc.; drop via SQL `DROP OWNED BY x; DROP ROLE x;` / `DROP DATABASE`). Adding `{name: postgres}` materializes the postgres superuser secret. Percona `spec.autoCreateUserSchema: true` default (annotation `postgres-operator.crunchydata.com/autoCreateUserSchema` upstream).
- **Secret** `<cluster>-pguser-<user>` keys: `user`, `password`, `verifier` (SCRAM-SHA-256), `host`, `port`, `dbname` (first DB), `uri`, `jdbc-uri`, plus `pgbouncer-host/port/uri/jdbc-uri` when pooler enabled (Crunchy; present in Percona fork too).
- **Rotate to generated password**: `kubectl patch secret <cluster>-pguser-<user> -p '{"data":{"password":""}}'` (operator regenerates password+verifier immediately). **Set custom password**: `kubectl patch secret <...> -p '{"stringData":{"password":"<pw>","verifier":""}}'` (operator computes verifier; change is applied when verifier changes).
- **Superusers via pgBouncer** blocked by default; Percona `spec.proxy.pgBouncer.exposeSuperusers: true` to allow.
- **Risk**: low-medium (SUPERUSER grants; app outage if rotated password not propagated — pooler creds auto-updated by operator).
- **Declarative**: yes except drops (SQL).
- **Completion**: secret data updated (new `resourceVersion`, verifier present); test login `kubectl exec ... -- psql "postgresql://u:p@<c>-primary/<db>" -c 'select 1'`.
- **Live signals**: secret watch, `pg_roles` query, pgBouncer auth errors in logs.

## 8. Scale replicas horizontally (Tier B)
- **Mechanics**: `spec.instances[i].replicas` (both). `kubectl patch pg/<c> --type=json -p '[{"op":"replace","path":"/spec/instances/0/replicas","value":3}]'`. New members bootstrap via Patroni `create_replica_methods` = `[basebackup, pgbackrest]`; after first auto full-backup succeeds pgbackrest is prepended (delta restore from repo — much faster/cheaper than basebackup for big DBs). Multiple named instance sets supported (`spec.instances[].name` — distinct STS groups, e.g. for different AZs/resources).
- **Scale-down caveats**: operator picks victims (never the primary without switchover); PDB `spec.instances[i].minAvailable` (default 1 when replicas>1) blocks eviction-based disruption but not spec-driven scale-down; removed members' PVCs are deleted; if synchronous replication was configured, reducing below sync quorum blocks writes.
- **Risk**: up=low (I/O load during clone); down=medium.
- **Declarative**: yes.
- **Completion**: Percona `status.postgres.instances[i].ready == size` & `state=ready`; Crunchy `status.instances[i].readyReplicas == replicas`; new member `streaming` with Lag 0 in patronictl.
- **Live signals**: pod scheduling events, basebackup/pgbackrest-restore progress in new pod's `database` container log, `pg_stat_replication` row count, network/IO on primary, lag of new member.

## 9. Vertical resize CPU/memory (Tier B)
- **Mechanics**: `spec.instances[i].resources.{requests,limits}.{cpu,memory}` (K8s ResourceRequirements; limit-only ⇒ request defaults to limit). Also `proxy.pgBouncer.resources`, `backups.pgbackrest.repoHost.resources`, `backups.pgbackrest.jobs.resources`, sidecar containers (`instances[i].containers.replicaCertCopy.resources` etc. in Percona, `pmm.resources`). Pod-template change ⇒ operator rolling recreation (replicas first, primary last ⇒ one switchover).
- **Risk**: medium-high — lowering memory below `shared_buffers`+work_mem footprint ⇒ OOMKilled CrashLoop; scheduler may fail to place larger pods (Pending). Preflight: node allocatable check + `shared_buffers` sanity.
- **Declarative**: yes. **Completion**: `updatedReplicas==replicas` / state ready; verify `kubectl get pod -o jsonpath='{.spec.containers[?(@.name=="database")].resources}'`.
- **Live signals**: rolling progress, Pending-pod events (`FailedScheduling`), OOMKilled restart counts, kubectl top pods, harness TPS.

## 10. Backup schedules & retention (Tier B)
- **Schedules**: `spec.backups.pgbackrest.repos[i].schedules.{full,differential,incremental}` = cron strings (CRD min length 6; standard 5-field cron). Operator materializes CronJobs `<cluster>-repoN-{full,diff,incr}` (label `.../pgbackrest-cronjob`). Percona: each scheduled run also creates a `pg-backup` object.
- **Retention**: `spec.backups.pgbackrest.global`: `repoN-retention-full: "14"`, `repoN-retention-full-type: count|time` (time=days), `repoN-retention-diff: "3"`. Expiry runs after each backup; expiring a full expires dependent diff/incr + WAL.
- **Pause schedules**: remove the schedule key (declarative). Patching CronJob `spec.suspend=true` directly gets reverted on reconcile — don't.
- **Risk**: low (retention mistakes can silently delete recovery points — show effective policy).
- **Completion**: CronJob exists/updated, `status.pgbackrest.scheduledBackups[]` entries with `completionTime`,`succeeded`; `pgbackrest info` backup list.
- **Live signals**: next-run countdown (`kubectl get cronjob -o jsonpath'{.status.lastScheduleTime}'` + schedule), running Job, repo disk trend.

## 11. Minor version upgrade / image bump (Tier C)
- **Percona**: patch images together (single merge patch): `spec.image` (`percona/percona-distribution-postgresql:<17.x>`), `spec.proxy.pgBouncer.image`, `spec.backups.pgbackrest.image`, optionally `spec.pmm.image`; bump `spec.crVersion` only when the **operator** itself was upgraded first (order: operator ➝ crVersion ➝ images; check compat at `https://check.percona.com/versions/v1/pg-operator/<ver>`). Rollout is automatic rolling (replicas→primary; one switchover).
- **Crunchy**: bump `spec.image` (or rely on operator RELATED_IMAGE_* defaults after operator upgrade); `spec.postgresVersion` stays (major only).
- **Risk**: medium — brief failover; **UBI9 glibc collation change (Percona ≥2.6)**: collation version mismatch can corrupt text-index ordering ⇒ post-upgrade `REINDEX` of affected indexes / `ALTER COLLATION ... REFRESH VERSION` check. No rollback of PG binaries once WAL format written by newer minor (usually safe, but treat as forward-only).
- **Declarative**: yes.
- **Completion**: every pod's image == target (`kubectl get pods -l .../cluster=<c> -o jsonpath='{..containers[?(@.name=="database")].image}'`), all Ready, `SELECT version();` per member, state ready.
- **Live signals**: rolling progress (which member is restarting), patronictl leader changes, image pull events, harness error window.

## 12. Storage / PVC expansion (Tier C, urgent when WAL fills)
- **Preflight (both)**: `kubectl get sc <class> -o jsonpath='{.allowVolumeExpansion}'` must be `true`; K8s cannot shrink PVCs.
- **Percona ≥2.5.0**: edit `spec.instances[i].dataVolumeClaimSpec.resources.requests.storage` — operator patches PVCs itself. Auto-grow: operator env `PGO_FEATURE_GATES="AutoGrowVolumes=true"` + set cap `spec.instances[i].dataVolumeClaimSpec.resources.limits.storage` → operator expands when usage crosses threshold.
- **Crunchy v5**: same path `spec.instances[i].dataVolumeClaimSpec.resources.requests.storage`; conditions `PersistentVolumeResizing` → in progress, `PersistentVolumeResizeError` → failed (e.g. SC refuses). Also expandable: `instances[i].walVolumeClaimSpec`, `instances[i].tablespaceVolumes[].dataVolumeClaimSpec`, `backups.pgbackrest.repos[i].volume.volumeClaimSpec` (repo volume!).
- **allowVolumeExpansion=false workaround**: add a new instance set with bigger `dataVolumeClaimSpec`, wait for replicas to stream, switchover, remove old set (works because members rebuild from primary/repo).
- **Risk**: medium; some CSI drivers require pod restart for FS resize (PVC condition `FileSystemResizePending`), most do online expansion.
- **Declarative**: yes. **Completion**: `kubectl get pvc -l .../cluster=<c>` `status.capacity.storage` == requested; resizing condition cleared; `df -h /pgdata` inside pod.
- **Live signals**: PVC events (`ExternalExpanding`, `FileSystemResizeSuccessful`), df% inside pods, WAL dir size (`du -sh /pgdata/pg17_wal`).

## 13. Connection pooler ops (Tier C)
- **Mechanics**: `spec.proxy.pgBouncer.replicas` (default 1) — scale like any deployment via CR patch; **0 = pause pooler** (endpoints drain; direct `-primary` service still works). Config: `spec.proxy.pgBouncer.config.global.{pool_mode,max_client_conn,default_pool_size,...}` map (pgbouncer.ini [pgbouncer] section) — change triggers pooler rollout; `port` (default 5432, changing restarts postgres too b/c secrets embed it), `customTLSSecret`, `expose.type: LoadBalancer` + `expose.annotations/labels`, `minAvailable` (PDB, IntOrString), `exposeSuperusers` (Percona). Crunchy path identical minus exposeSuperusers.
- **Risk**: low-medium (bad ini value ⇒ pgbouncer CrashLoop; app traffic through pooler).
- **Declarative**: yes. **Completion**: Crunchy condition `ProxyAvailable=True`; Percona `status.pgbouncer.ready==size`; deployment `<c>-pgbouncer` readyReplicas.
- **Live signals**: pgbouncer pod logs, `SHOW STATS/POOLS/CLIENTS` via admin console (exec psql to pgbouncer pod port 5432 db `pgbouncer` — needs stats_users), connection counts on `pg_stat_activity` (`usename='_crunchypgbouncer'`).

## 14. Node maintenance: cordon/drain (Tier C)
- **Flow**: (1) `kubectl get pods -o wide -L postgres-operator.crunchydata.com/role -l .../cluster=<c>` to map members→nodes; (2) if primary on target node → switchover away (op 6); (3) `kubectl cordon <node>`; (4) `kubectl drain <node> --ignore-daemonsets --delete-emptydir-data`; PDBs (instances `minAvailable` default 1 when replicas>1; pgBouncer `minAvailable`) make drain block rather than take the last healthy member — surface "drain blocked by PDB" as a signal, not an error; (5) `kubectl uncordon`; (6) verify patronictl all streaming.
- **Placement controls to expose**: `spec.instances[i].affinity.podAntiAffinity` (Percona example uses preferred anti-affinity on `.../instance-set`), `instances[i].topologySpreadConstraints.{maxSkew,topologyKey,whenUnsatisfiable,labelSelector}`, `tolerations`, `priorityClassName`; same knobs for `repoHost` and `pgBouncer`.
- **Risk**: medium. **Declarative**: partially (placement yes; drain imperative). **Completion**: node unschedulable flag; all cluster pods Running on other nodes; lag 0. **Live signals**: eviction events, Pending pods, patronictl.

## 15. Clone to new cluster / dataSource (Tier C)
- **Percona/Crunchy identical concept**: create a **new** cluster CR with `spec.dataSource`:
  - `dataSource.postgresCluster: {clusterName, clusterNamespace (cross-ns needs cluster-wide operator), repoName, options[]}` — clone from live cluster's repo. PITR clone: add `options: ["--type=time","--target=\"2025-11-30 15:12:11+03\"", "--set=<baseBackupID>"]`; selective DB: `--db-include=app1`.
  - `dataSource.pgbackrest: {stanza: db, configuration:[{secret:{name:<creds>}}], global:{repo1-path: <source path>}, repo:{name: repo1, s3|gcs|azure:{...}}}` — restore from cloud repo when source cluster is gone. New cluster's own `backups.pgbackrest.global.repo1-path` MUST differ from source path to avoid clobbering source backups.
  - Precedence: `postgresCluster` wins if both set. `dataSource` is only honored at cluster creation.
- **Risk**: low (does not touch source; watch S3 egress/IO).
- **Declarative**: fully. **Completion**: Crunchy condition `PostgresDataInitialized=True`, then ready counts; Percona `status.state=ready`, `kubectl describe pg cluster2` ready==size. **Live signals**: restore Job logs (label `.../pgbackrest-restore`), bytes restored (pgbackrest log), new-cluster pod phases.

## 16. Restore in-place & PITR (Tier D — destructive)
- **Percona**: create `PerconaPGRestore`: `{spec:{pgCluster, repoName, options[]}}`. Variants: no options = latest full + replay all WAL; `--type=immediate --set=<backupID>` = exactly that backup; `--type=time --target="YYYY-MM-DD HH:MM:SS+TZ"` (+ optional `--set`) = PITR. Get target candidates: `kubectl get pg-backup <b> -o jsonpath='{.status.latestRestorableTime}'`; backup IDs via `pgbackrest --stanza=db info`. Track `kubectl get pg-restore` → `Starting/Running/Succeeded/Failed`. **Stop a stuck restore**: `kubectl annotate pg/<c> postgres-operator.crunchydata.com/pgbackrest-restore-` (removes annotation Percona set under the hood).
- **Crunchy**: `spec.backups.pgbackrest.restore: {enabled: true, repoName, options[]}` then trigger `kubectl annotate --overwrite postgrescluster/<c> postgres-operator.crunchydata.com/pgbackrest-restore="$(date)"`; after success set `enabled:false`. Status: `status.pgbackrest.restore{id,finished,succeeded}` + `PostgresDataInitialized` condition.
- **Semantics**: cluster is stopped, pgBackRest delta-restores the primary PVC, WAL replay to target, promote ⇒ **new timeline**; replicas re-created. Everything after target time is gone (in-place = overwrite).
- **Risk**: HIGH. Console must force typed confirmation + suggest pre-restore on-demand full backup + show target-time vs latestRestorableTime validation. Failure modes: bad options hang restore (fix: remove annotation), corrupted repo (delete CR + startup PVC, recreate). Post-major-upgrade first restore can hit `pg_rewind: could not find common ancestor` on replicas ⇒ reinit replica (op 19 note) or pre-set `spec.patroni.removeDataDirectoryOnDivergedTimelines: true` (data-loss tradeoff).
- **Declarative**: Percona yes (CR); Crunchy spec+annotation.
- **Completion**: pg-restore `Succeeded`; cluster `ready`; timeline TL bumped in patronictl; sanity query of restored data.
- **Live signals**: restore Job log tail (`%` progress lines from pgbackrest), cluster state (`initializing`), member roles, WAL replay LSN progression, events.

## 17. Pause / resume, shutdown, unmanaged (Tier D)
- **Percona**: `spec.pause: true` ⇒ graceful full stop (primary, replicas, pgBackRest, jobs; PVCs kept) — maps to Crunchy `spec.shutdown`. `spec.unmanaged: true` ⇒ operator stops reconciling entirely (maps to Crunchy `spec.paused`); no shutdown, probes still run. Verified mapping in source: `postgresCluster.Spec.Paused = cr.Spec.Unmanaged; postgresCluster.Spec.Shutdown = cr.Spec.Pause`.
- **Crunchy**: `spec.shutdown: true` (stop pods; former primary restarts first on resume via `status.startupInstance`); `spec.paused: true` (reconciliation freeze; only `Progressing` condition updates).
- **Blocker**: pause is refused while a backup Job runs — delete it first (`kubectl delete job -l postgres-operator.crunchydata.com/pgbackrest-backup`).
- **Risk**: high (full outage by design). **Declarative**: yes. **Completion**: Percona `status.state: paused` (via `stopping`); zero instance pods; resume → `ready`. **Live signals**: pod termination sequence, state field, events.

## 18. Certificate rotation (Tier D)
- **Operator-generated certs**: long-lived, auto-rotated — no action. **cert-manager**: Certificates renew automatically; operator hot-reloads secret contents.
- **Custom certs, same secret names**: regenerate files, `kubectl create secret generic <name> --from-file=tls.crt=... --from-file=tls.key=... --from-file=ca.crt=... --dry-run=client -o yaml | kubectl apply -f -` — operator detects data change and applies **without restart** (hot reload). Default secret names Percona: `<cluster>-cluster-cert`, `<cluster>-replication-cert`.
- **New secret names**: update `spec.secrets.customTLSSecret.name` / `spec.secrets.customReplicationTLSSecret.name` (Percona; Crunchy: top-level `spec.customTLSSecret` / `spec.customReplicationTLSSecret`) ⇒ cluster restart.
- **Root CA swap (Percona documented flow)**: create NEW secrets (never overwrite), `spec.pause=true`, patch `spec.secrets.customRootCATLSSecret{name, items[{key:ca.crt,path:root.crt},{key:ca.key,path:root.key}]}` + both TLS secrets, `spec.pause=false`.
- **Constraints**: server cert SAN must cover `<cluster>-primary.<ns>.svc`; replication cert CN must be `_crunchyrepl`; all signed by same CA.
- **Expiry check (console health)**: `kubectl get secret <s> -o jsonpath='{.data.tls\.crt}' | base64 -d | openssl x509 -noout -enddate`.
- **Risk**: high (mistake breaks replication + client TLS; `replication-cert-copy` container logs show failures). **Declarative**: yes. **Completion**: `openssl s_client` date check via exec, no cert errors in logs, all members streaming.

## 19. Vacuum / bloat management, pg_repack, reinit (Tier D)
- **Autovacuum tuning**: op 5. **Observation SQL**: `pg_stat_user_tables (n_dead_tup, last_autovacuum)`, `pg_stat_progress_vacuum`, bloat estimate queries; `pg_stat_activity WHERE query ILIKE 'autovacuum%'`.
- **Manual**: `kubectl exec <primary> -c database -- psql -d <db> -c 'VACUUM (VERBOSE, ANALYZE) <table>;'`.
- **pg_repack**: Percona ships it — enable `spec.extensions.builtin.pg_repack: true` (pod restart), `CREATE EXTENSION pg_repack;`, then run the bundled client inside the primary's database container: `kubectl exec <primary> -c database -- pg_repack -d <db> --table <t>` (needs ~2x table free disk; brief AccessExclusiveLock at swap; killable). Crunchy standard images: verify binary presence first (`command -v pg_repack`); otherwise run as a Job with a matching image.
- **Replica reinit** (bloat's cousin — divergent/corrupt replica): `kubectl delete pvc/<pod>-pgdata pod/<pod>` (operator rebuilds via create_replica_methods) or `kubectl exec ... -- patronictl reload <c>-ha <member>` then `patronictl reinit <c>-ha <member> --force`. Automation: `spec.patroni.removeDataDirectoryOnDivergedTimelines: true` (possible data loss — warn).
- **Risk**: medium. **Completion**: pg_repack exit 0; dead tuples reset; reinit → member back to `streaming` lag 0. **Live signals**: pg_stat_progress_vacuum, table size, replica rebuild log, patronictl.

## 20. Extension management (Tier D)
- **Percona built-ins**: `spec.extensions.builtin: {pg_stat_monitor, pg_audit, pgvector, pg_repack}` booleans (defaults in deploy/cr.yaml: pg_audit=true, pg_stat_monitor=false, pgvector=false, pg_repack=false) — apply ⇒ **pod restarts** (operator manages shared_preload_libraries for these). **Custom extensions (≥2.3)**: upload `<ext>-pg<major>-<ver>.tar.gz` (rooted at `usr/pgsql-<major>/{lib,share/extension}`) to S3-compatible storage; `spec.extensions.storage:{type:s3,bucket,region,endpoint,secret:{name}}` + `spec.extensions.custom:[{name,version}]` ⇒ operator downloads/installs, restarts pods; version bump = update `version`.
- **shared_preload_libraries** additions (pg_cron etc.): `spec.patroni.dynamicConfiguration.postgresql.parameters.shared_preload_libraries` — postmaster context ⇒ rolling restart. Then per-DB `CREATE EXTENSION <x>;` via psql (not automated).
- **Crunchy v5**: no download mechanism — extension must exist in image (crunchy images ship pgAudit, pg_partman, pg_cron, pgvector, pgnodemx, ...); enable via shared_preload_libraries + CREATE EXTENSION. `spec.databaseInitSQL` is bootstrap-only.
- **Major-upgrade interaction**: pgaudit must be dropped/recreated per DB after major upgrade; custom extensions need a pg<newmajor> package available.
- **Risk**: high blast radius via shared_preload_libraries typos (cluster-wide CrashLoop). **Declarative**: install yes; CREATE EXTENSION no. **Completion**: pods restarted, `SELECT * FROM pg_available_extensions WHERE name='x';`, `\dx`. **Live signals**: restart rollout, error log tail.

## 21. Monitoring integrations (Tier D setup, Tier A consumption)
- **Percona PMM**: `spec.pmm: {enabled: true, image: percona/pmm-client:<3.x>, serverHost: <pmm-svc>, secret: <cluster>-pmm-secret, customClusterName, querySource, resources}`; secret key **`PMM_SERVER_TOKEN`** (PMM3 service-account token) or `PMM_SERVER_KEY` (PMM2 API key, deprecated); toggling restarts instance pods (adds `pmm-client` sidecar). Hash annotation `pgv2.percona.com/monitor-user-secret-hash` tracks monitor user changes.
- **Crunchy exporter**: `spec.monitoring.pgmonitor.exporter: {image, configuration: [projected volumes w/ queries.yml → extend.query-path], customTLSSecret, resources}` ⇒ `exporter` container (postgres_exporter, port 9187), label `.../crunchy-postgres-exporter=true` for scrape discovery; annotation `postgres-operator.crunchydata.com/postgres-exporter-collectors: None` disables default collectors. pgMonitor metric families worth charting: `ccp_connection_stats_*`, `ccp_database_size_bytes`, `ccp_replication_lag_size_bytes` / `ccp_replication_lag` (seconds), `ccp_wal_activity_total_size_bytes`, `ccp_backrest_last_full_backup_time_since_completed_seconds` (+diff/incr), `ccp_locks_count`, `ccp_transaction_wraparound_percent_towards_wraparound`, plus standard `pg_stat_database_xact_commit/rollback` rates, `pg_stat_activity_count`, `pg_stat_bgwriter_*`, `pg_replication_slots_*`.
- **Console live-chart shortlist for any op**: TPS + latency (from pgbench harness), connections, replication lag bytes/sec per member, WAL generation rate, archiver failed_count, PVC used %, CPU/mem (kubectl top pods), pod restarts, role changes, backup job durations.
- **Risk**: low (restarts on enable). **Declarative**: yes. **Completion**: sidecar present in pods, `curl localhost:9187/metrics` via exec, PMM inventory shows node.

## 22. Major version upgrade (Tier D)
- **Percona (`PerconaPGUpgrade`, tech-preview since 2.6, GA 2.9.0)**: apply CR:
```yaml
apiVersion: pgv2.percona.com/v2
kind: PerconaPGUpgrade
metadata: {name: cluster1-17-to-18}
spec:
  postgresClusterName: cluster1
  image: percona/percona-distribution-postgresql-upgrade:<ver>
  fromPostgresVersion: 17   # CRD bounds: from 12–17
  toPostgresVersion: 18     # to 13–18
  toPostgresImage: percona/percona-distribution-postgresql:<18.x>
  toPgBouncerImage: percona/percona-pgbouncer:<ver>
  toPgBackRestImage: percona/percona-pgbackrest:<ver>
```
  The operator then **automatically**: pauses cluster (downtime!), annotates cluster `pgv2.percona.com/allow-upgrade: <upgradeName>`, runs pg_upgrade Jobs (data duplicated on each PVC — **needs ≥2x free space**), resumes with new version. Cleanup: exec `rm -rf /pgdata/pg17 /pgdata/pg17_wal`; `kubectl delete perconapgupgrade <name>` removes Jobs/Pods. If it fails, cluster stays paused — unpause manually to come back on old version.
- **Crunchy (`PGUpgrade`, manual orchestration)**: 1) create PGUpgrade `{postgresClusterName, fromPostgresVersion, toPostgresVersion, image}`; 2) `kubectl annotate postgrescluster <c> postgres-operator.crunchydata.com/allow-upgrade="<pgupgrade-name>"`; 3) `spec.shutdown: true`; 4) wait for upgrade Jobs (PGUpgrade conditions `Progressing` → `Succeeded`); 5) set `spec.postgresVersion: <new>` and `shutdown: false`; 6) remove annotation. Take a fresh backup immediately (new stanza/timeline).
- **Post-steps (both)**: `DROP EXTENSION pgaudit; CREATE EXTENSION pgaudit;` per DB; run `ANALYZE`/`vacuumdb --analyze-in-stages` (pg_upgrade drops planner stats); collation/REINDEX check; first-restore-after-upgrade pg_rewind "no common ancestor" pitfall (see op 16).
- **Risk**: HIGHEST (full downtime + disk + irreversible after new-version writes). **Declarative**: Percona 1-CR; Crunchy multi-step. **Completion**: PGUpgrade condition `Succeeded=True`; cluster ready; `status.postgres.version` / `SELECT version();`; pgBackRest stanza-upgrade done (`pgbackrest info` shows new db version). **Live signals**: upgrade Job pod logs (pg_upgrade output), cluster state paused→initializing→ready, PVC free-space during copy.

## 23. Standby cluster & DR promotion (Tier D)
- **Create standby**: new cluster CR with `spec.standby: {enabled: true, repoName: repo1}` (repo-based: replays WAL from shared pgBackRest repo) and/or `{host: <primary-svc-or-lb>, port}` (streaming), optional Percona `standby.maxAcceptableLag: 10Mi` (lag breach ⇒ primary-standby pod unready, cluster `initializing`, condition `StandbyLagging`).
- **Promote**: set `spec.standby.enabled: false`. **MUST ensure old primary is shut down/deleted first — both writing to one repo = split-brain WAL corruption.** Recovery from split-brain: keep one primary, stop the other, take fresh full backup.
- **Risk**: high. **Declarative**: yes. **Completion**: `pg_is_in_recovery()` = false, state ready, writes accepted. **Live signals**: standby lag vs source, timeline, repo WAL archive activity.

## 24. Troubleshooting flows (console "diagnose" buttons)
- **Instance pod CrashLoopBackOff**: `kubectl describe pod` (last state, exit code: 137=OOM ⇒ raise memory / lower shared_buffers); `kubectl logs -p -c database`; if postgres itself won't start, use the **sleep-forever hatch**: `kubectl exec <pod> -- touch /pgdata/sleep-forever; kubectl delete pod <pod>` → pod comes up idle ("node entered an infinite sleep") → exec in, inspect `/pgdata/pg17/log`, run `pg_ctl -D /pgdata/pg17 start` manually; remove file when done. Diverged timeline ⇒ reinit (op 19).
- **WAL-full PVC**: detect `df /pgdata` ≥90%, `du /pgdata/pg17_wal`, `pg_stat_archiver.failed_count` rising, inactive slots. Fix order: repair archiving (repo-host PVC full ⇒ expand `repos[].volume`; S3 creds; check backup-job/pgbackrest logs) → expand data PVC (op 12) → drop inactive slot (`SELECT pg_drop_replication_slot('x');`). Last resort `patronictl edit-config --pg archive_command=/bin/true` (breaks PITR chain — full backup required after). Never `rm` WAL segments.
- **Failed backup job**: `kubectl get jobs -l postgres-operator.crunchydata.com/pgbackrest-backup`; `kubectl logs job/<j>`; Percona `kubectl get pg-backup` state Failed + `kubectl describe pg-backup`. Common causes: bad storage secret, repo-host TLS cert mismatch, stanza mismatch after restore, repo disk full. Retry: Percona delete+recreate pg-backup; Crunchy re-annotate with a **new** value (and delete the failed Job). Debug flag `--log-level-stderr=debug` in options.
- **Stuck leader election / no primary**: `patronictl list` no Leader; DCS is Kubernetes (leader lock on `<cluster>-ha` Endpoints — `kubectl get endpoints <c>-ha -o yaml` annotations show leader+ttl); check if Patroni paused (`patronictl list` footer "maintenance mode" ⇒ `patronictl resume <c>-ha`); force: `patronictl failover <c>-ha --candidate <member> --force`; inspect database container logs for DCS errors (RBAC, API throttling).
- **Replica not streaming**: patronictl State `in archive recovery` (WAL-shipping only — streaming broken) or `crashed`; check `pg_stat_wal_receiver` on replica, `pg_stat_replication` on primary (row missing?); `replication-cert-copy` container logs (cert auth); `pg_rewind` errors ⇒ reinit; verify `<c>-pods`/`-ha` service DNS.
- **Operator not reconciling**: check `spec.unmanaged` (Percona) / `spec.paused` (Crunchy) not set; operator pod logs; `status.observedGeneration` lagging `metadata.generation`; CR events via `kubectl describe`.
